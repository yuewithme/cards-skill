# Mode: Method Intro

Use this mode when the user asks to introduce, explain, teach, summarize, or visualize a method, tool, model, framework, or analysis approach.

## Goal

Create a card that helps the reader understand:

1. What the method is.
2. What problem it solves.
3. How its internal structure works.
4. How to use it.
5. What its real output should be.

This is a teaching card, not a case analysis.

## Content Model

Extract or infer:

- **Method name**: short title, usually the largest semantic anchor.
- **One-sentence definition**: what the method does.
- **Core problem**: the question this method helps answer.
- **Structure**: dimensions, quadrants, steps, curve, matrix, funnel, chain, or forces.
- **Usage steps**: usually 3-5 actions.
- **True output**: what decision or judgment the method should produce.
- **Misuse warning**: optional, but useful when the method is often used as a checklist.

## Recommended Layouts

Choose by method shape:

| Method shape | Layout |
| --- | --- |
| 2x2 or four-quadrant tool | Matrix as central anchor |
| Five forces / five dimensions | Radial map or asymmetric factor grid |
| Process / sequence | Timeline or vertical pipeline |
| Value chain / curve | Large curve or chain diagram |
| Nine-cell canvas or large matrix | Wide 1280px canvas with the original framework geometry preserved |
| Checklist-like model | Grouped field manual layout |
| Abstract concept | Giant word or symbol + supporting blocks |

## Required Sections

Include these sections unless the content clearly calls for a different structure:

1. **Header**
   - REF code, title, and one-sentence definition.
   - Avoid a centered hero unless the concept is sparse and iconic.

2. **Framework Body**
   - The main visual explanation.
   - Make the method structure visible before adding commentary.
   - Examples: SWOT 2x2, STEEP factor grid, smile curve, five forces map.

3. **How To Use**
   - 3-5 concrete usage steps.
   - Use verbs: define, scan, sort, judge, translate, prioritize.

4. **True Output**
   - A strong statement that tells the reader what the method should produce.
   - Avoid vague phrases such as "improve understanding" or "empower decisions".

5. **Bottom Thesis**
   - One sentence that corrects a common misunderstanding or lands the method's value.

## Writing Rules

- Explain the method in plain Chinese.
- Do not over-explain history unless the user asks.
- Prefer "它帮助你判断..." over "它是一种用于..." when possible.
- Avoid decorative fake examples.
- Make every label actionable or explanatory.
- If the method acronym has letters, define each letter clearly.

## Visual Rules

- Create one dominant visual anchor.
- Use the selected content geometry; do not force every method into cards.
- Preserve the real geometry of canvas tools. For Business Model Canvas and similar frameworks, make the central canvas wide enough that each cell has room for a label, question, and interpretation.
- Do not let decorative marks cover explanatory text.
- Keep body text readable after export at 1280px width.
- Do not add footer, logo, author signature, or default source line.

## Quality Check

Before rendering, verify:

- The reader can identify the method in 2 seconds.
- The reader can explain the method's structure in 10 seconds.
- The "how to use" section contains actions, not nouns.
- The "true output" section names a decision, judgment, or prioritization.
- The design would not still make sense if the method name were swapped with a different method.
